# Проект расширенного тактического анализа пляжного волейбола

## Обзор проекта

Этот проект предоставляет автоматизированное решение для извлечения тактических данных, связанных с подачей, из видеозаписей матчей по пляжному волейболу. Используя продвинутые методы компьютерного зрения (треккинг объектов и оценку позы на базе YOLOv8), система не только точно определяет события подачи, но и количественно оценивает **тип подачи** и **зону приземления**.

Ключевая ценность инструмента - его **стабильность** и **настраиваемость**. После нескольких итераций текущий алгоритм основан на физической интуиции и проверке на реальных сценариях, что делает его надежным для тренеров, игроков и аналитиков данных.

---

## Основные возможности

- **Высокоточное обнаружение подачи**: используется проверенная и стабильная четырехэтапная машина состояний (`search toss` -> `confirm toss` -> `wait apex` -> `wait hit`), чтобы снизить число ложных срабатываний из-за падения мяча под действием гравитации или движений игроков.

- **Точная идентификация подающего**:
  - **Метод отката по кадрам (backtracking)**: после первичного обнаружения удара система отматывает несколько кадров назад (`--search_offset`) и определяет подающего в оптимальный момент, когда расстояние между игроком и мячом минимально.
  - **Алгоритм минимального расстояния от точки до прямоугольника**: снижает ошибки, связанные с перспективой, измеряя кратчайшее расстояние от точки до границы прямоугольника, что улучшает сопоставление игрока и мяча.

- **Продвинутый анализ типа подачи (прыжковая / подача с места)**:
  - **Анализ траектории позы**: извлекаются ключевые точки тела с помощью `yolov8n-pose.pt`.
  - **Метод смещения в одном окне**: используется вертикальное смещение бедер перед подачей и вычисляется расстояние от самой низкой точки приседа до самой высокой точки прыжка, чтобы устойчиво классифицировать тип подачи даже при периодических ошибках детекции по кадрам.

- **Анализ зоны приземления подачи (A/B/C)**:
  - Требуется файл `court_config.json`, задающий четыре угла площадки.
  - Программа автоматически делит заднюю линию на три области и классифицирует зону приземления как A, B или C по позиции подающего.

- **Сильные инструменты визуальной отладки**:
  - **График траектории прыжковой подачи**: с `--debug_jump_serve` программа генерирует график высоты бедер для каждого решения по прыжковой подаче.
  - **Сохранение ключевых кадров**: автоматически сохраняются три кадра до и после каждого обнаруженного события подачи для быстрой ручной проверки.

- **Автоматический экспорт данных**:
  - После анализа результаты по всем видео (имя сегмента, зона подачи, тип подачи) агрегируются в `analysis_summary.csv`, готовый для Excel и других инструментов анализа.

- **Управляемая параллельная обработка**:
  - Параметр `--workers` задает количество одновременно обрабатываемых видео и позволяет балансировать скорость выполнения и стабильность работы железа, избегая исчерпания памяти.

## Установка и требования

1. **Окружение Python**: рекомендуется Python 3.8+.

2. **Conda (рекомендуется)**:
   ```bash
   conda create -n beach-volleyball-tracker python=3.8
   conda activate beach-volleyball-tracker
   ```

3. **Зависимости**:
   ```bash
   pip install opencv-python numpy tqdm ultralytics matplotlib
   ```
   *(Пакет `ultralytics` автоматически устанавливает `torch` и связанные зависимости.)*

4. **Файлы моделей**:
   - Убедитесь, что `video_processing/track_ball_and_player.py` может получить доступ к выбранным файлам моделей, например `yolov8s-pose.pt` и `ball_best.pt`.
   - Если используется последняя версия предоставленного скрипта трекинга, убедитесь, что `yolov8n.pt` и `yolov8n-pose.pt` доступны локально или могут быть автоматически загружены.

## Руководство по использованию

Процесс состоит из двух основных шагов:

### Шаг 1: Задать границы площадки (один раз для каждого типа камеры/ракурса)

Чтобы включить анализ зон подачи, создайте конфигурационный файл с координатами четырех углов площадки.

1. **Запустите генератор конфигурации**:
   ```bash
   python court_definition/court_config_generator.py --video_path "path/to/your/sample_video.mp4"
   ```

2. **Отметьте углы площадки**:
   - Программа показывает первый кадр.
   - Кликните четыре угла строго в этом порядке: `top-left -> bottom-left -> bottom-right -> top-right`.
   - После завершения `court_config.json` будет сохранен в корне проекта.

### Шаг 2: Запустить основной скрипт анализа

Это ключевой шаг для анализа всех видео и генерации выходных файлов.

1. **Базовая команда**:
   ```bash
   python run_analysis_all_in_one.py --input_folder "your_video_folder" --court_config "court_config.json"
   ```

2. **Полный режим отладки (рекомендуется для первых запусков или настройки параметров)**:
   ```bash
   python run_analysis_all_in_one.py --input_folder "your_video_folder" --court_config "court_config.json" --workers 1 --debug_jump_serve --overwrite
   ```
   Это включает графики отладки прыжковой подачи и принудительно использует один воркер для стабильности.

## Параметры командной строки

Основной скрипт `run_analysis_all_in_one.py` предоставляет богатый набор параметров для настройки.

### Основные параметры

| Параметр | Обязательный | По умолчанию | Описание |
| :--- | :---: | :---: | :--- |
| `--input_folder` | **Да** | - | Путь к папке с видео для анализа. |
| `--court_config` | Нет | `None` | Путь к `court_config.json`. При указании включает анализ зон подачи. |
| `--workers` | Нет | `2` | Количество видео, обрабатываемых параллельно. Настраивается под RAM/VRAM. |
| `--overwrite` | Нет | `False` | Повторно анализировать все видео и перезаписывать старые результаты. |
| `--debug_jump_serve` | Нет | `False` | Включить режим визуальной отладки прыжковой подачи (графики траектории бедер). |

### Параметры основного алгоритма (стабильная версия v11)

| Параметр | По умолчанию | Описание |
| :--- | :---: | :--- |
| `--hit_v` | `40.0` | Минимальная мгновенная полная скорость для детекции удара. |
| `--toss_vy` | `8.0` | Минимальная начальная вертикальная скорость вверх для запуска гипотезы подброса. |
| `--vertical_ratio` | `1.5` | Во время подброса вертикальная скорость должна быть как минимум во столько раз больше горизонтальной. |
| `--hit_h_ratio` | `2.5` | **Важно**: максимальное отношение горизонтальной/вертикальной скорости в момент удара; используется для фильтрации чисто вертикальных падений. |
| *(... другие параметры машины состояний, например `max_frames_to_apex` ...)* | ... | Параметры для настройки временных окон машины состояний. |

### Параметры расширенного анализа

| Параметр | По умолчанию | Описание |
| :--- | :---: | :--- |
| `--search_offset` | `3` | Количество кадров для отката от первичной точки удара при поиске подающего. |
| `--ball_leave_threshold` | `30` | **Устарел**: параметр, использовавшийся старой логикой точного тайминга. |
| `--jump_height_threshold` | `20` | Минимальное вертикальное смещение бедер (в пикселях) для классификации прыжка. |

## Описание выходных данных

После анализа все результаты сохраняются в папке, указанной через `--output_folder`.

- **`final_summary_refined/`**: содержит итоговые результаты анализа.
  - **`analysis_summary.csv`**: самый важный выходной файл; таблица для Excel с зоной и типом подачи по каждому сегменту.
  - **`summary_report.txt`**: текстовый сводный отчет, включающий статус отладки.
  - **`..._jump_debug_plot.png`**: (только в режиме отладки) график траектории высоты бедер для валидации прыжковой подачи.

- **`key_frames_for_review_refined/`**: хранит ключевые кадры для **всех видео** (3 кадра до и после `HIT`) для быстрой ручной проверки.

- **`[video_name]/`**: папка для каждого видео.
  - `tracking_output/`: сырые данные трекинга объектов (`..._all_frames_data_with_pose.json`).
  - `key_frames/`: ключевые кадры для конкретного видео.

## Устранение неполадок

- **Неточная детекция подачи (падающий мяч ошибочно классифицируется как удар)**:
  - Это ключевая проблема, решенная в `v11`. Попробуйте уменьшить `--hit_h_ratio` (например, с `2.5` до `1.5` или `1.0`), чтобы удар обязательно включал более выраженное горизонтальное движение.

- **Некорректная классификация прыжковая/с места**:
  1. Повторно запустите с `--debug_jump_serve`.
  2. Проверьте сгенерированный `..._jump_debug_plot.png`.
  3. Посмотрите `Displacement` в заголовке графика. Если очевидная прыжковая подача имеет смещение чуть ниже порога (по умолчанию `20`, например `18.5`), уменьшите `--jump_height_threshold`.

- **Программа падает с ошибками памяти (`CUDA error` или `MemoryError`)**:
  - Слишком много видео обрабатывается параллельно. Уменьшите `--workers`, начиная с `--workers 1`.
